# Phase 11 — Self-Service Cloud Provider Accounts

Project: Cloud Usage Monitoring and AI-Driven Predictive Resource Optimization Platform for Microservices
Phase: 11 (post-completion feature addition, beyond the original 10-phase plan)
Status: **Complete and verified**

---

## 1. Overview

Following the original 10-phase build, this phase adds a user-facing feature
requested directly: the ability for any user to configure their own cloud
provider account(s) through the UI, for **any** cloud provider (not a
hardcoded AWS/Azure/GCP list), with **no restriction on how many accounts**
a user may register, and each account scoped to a specific **region**.

## 2. Objectives Completed

- [x] `CloudProviderAccount` domain model, self-service (any authenticated user manages only their own), no count limit anywhere in the implementation
- [x] Credentials stored encrypted at rest (Fernet, via a dedicated `CLOUD_CREDENTIALS_ENCRYPTION_KEY` secret - deliberately separate from the JWT `SECRET_KEY`) and never returned by any read endpoint - write-only
- [x] Full CRUD REST API (`/api/v1/cloud-provider-accounts`), following the same ownership pattern already established by the Notifications endpoints
- [x] Frontend "Cloud Accounts" page: list, add, edit, delete, with a dynamic key/value credential editor (since different providers need different credential fields) and a provider dropdown that accepts any custom provider name via an "Other" option
- [x] 18 new backend tests (15 API + 3 crypto unit tests), including an explicit test creating 15 accounts for one user to verify the "no restriction on count" requirement directly
- [x] Alembic migration autogenerated and applied against the live database
- [x] Full live verification through the real frontend nginx proxy (not just the backend directly), including a custom, non-AWS/Azure/GCP provider name

## 3. Architecture Decisions

### Free-text `provider`, not an enum
The requirement was explicitly "any cloud provider," so `provider` is a
plain string column, validated only for length, not against a fixed list.
The frontend offers AWS/Azure/GCP as one-click choices plus an "Other"
option accepting any free-text provider name, satisfying the requirement
without the backend ever needing to know about a new provider in advance.

### Generic `credentials: dict[str, str]`, not provider-specific fields
An AWS account needs an access key ID and secret; a GCP account typically
needs a single service-account JSON blob; other providers need other
shapes entirely. Rather than modelling every provider's credential shape
individually, `credentials` is an arbitrary key/value map, encrypted as a
whole. This is what actually makes "any cloud provider" true at the data
layer, not just at the UI dropdown layer.

### Self-service ownership, no role restriction, no count limit
Every other domain resource in this platform (projects, deployments, etc.)
uses operator/admin role gates for writes, because those are shared
organizational resources. A user's own cloud provider credentials are not
shared - they belong to that user alone - so this feature instead follows
the pattern already established by the Notifications endpoints: any
authenticated user manages their own records, with an explicit ownership
check (`403 NOT_YOUR_CLOUD_ACCOUNT`) rather than a role check. No limit on
the number of accounts is enforced anywhere in the repository, service, or
database schema - this was verified directly by a test that creates 15
accounts for one user in a single test run.

### A dedicated encryption key, not reusing the JWT signing secret
Cloud credentials must be recoverable (to actually use them later), unlike
passwords (one-way hashed). `CLOUD_CREDENTIALS_ENCRYPTION_KEY` is a new,
separate secret from `SECRET_KEY`, so that rotating the JWT signing key
can never accidentally break decryption of stored cloud credentials, or
vice versa - two different cryptographic purposes should never share one
key. Since Fernet requires a specific 32-byte key format, an arbitrary
string secret (consistent with every other secret in this project) is
deterministically stretched into a valid Fernet key via SHA-256, so no
special key-generation tooling is needed to configure it.

## 4. API Endpoints

`POST /api/v1/cloud-provider-accounts` - create (self)
`GET /api/v1/cloud-provider-accounts?provider=&page=&page_size=` - list own, paginated, filterable
`GET /api/v1/cloud-provider-accounts/{id}` - get own
`PUT /api/v1/cloud-provider-accounts/{id}` - update own (credentials optional - omitting them leaves the stored value unchanged)
`DELETE /api/v1/cloud-provider-accounts/{id}` - delete own

All require authentication only, no role beyond `viewer`. Response bodies
never include `credentials` or `credentials_encrypted`.

## 5. Verification Results

- Backend test suite: **139/139 passing** (121 existing + 18 new)
- Alembic migration autogenerated cleanly against the live dev database and applied; `DESCRIBE cloud_provider_accounts` confirmed the resulting schema matches the model exactly
- Frontend build (`tsc -b && vite build`): zero errors
- Live end-to-end verification through the real nginx-proxied frontend (`http://localhost:3000/api/...`, not the backend directly): created two accounts for one user - one `aws` in `us-east-1`, one a fully custom `my-custom-cloud` provider in `eu-west-2` - both listed correctly, confirming the "any provider" and per-account "region" requirements are genuinely satisfied, not just accepted by the schema

## 6. Known Limitations (disclosed, not hidden)

- **No visual browser confirmation of the new UI page** - consistent with the disclosed limitation from Phase 7 (no headless browser tool available in this environment); verification here was via direct API calls through the real nginx proxy, plus a clean TypeScript build, not a rendered screenshot.
- **Credentials are stored encrypted but not validated against the actual cloud provider** - the platform does not attempt to call AWS/Azure/GCP APIs to confirm a submitted credential is genuine; that is a natural next step (e.g. a "test connection" action) but was out of scope for this request.
