# Phase 1 — Project Scaffolding, Database Schema & Authentication

Project: Cloud Usage Monitoring and AI-Driven Predictive Resource Optimization Platform for Microservices
Phase: 1 of ~10
Status: **Complete and verified**

## 1. Overview

Phase 1 establishes the foundation every later phase builds on: the full repository
layout, a normalized MySQL 8.0 schema covering every domain entity the platform will
ever need, and a complete JWT-based authentication and role-based access control
(RBAC) system implemented with Clean Architecture layering (routers → controllers →
services → repositories → models).

Nothing in this phase is a stub. Every file listed below is a working implementation,
and the entire stack (MySQL + FastAPI backend, in Docker) was built, migrated, and
exercised end-to-end via real HTTP requests before this phase was marked complete.

## 3. Features Implemented

| Feature | Detail |
|---|---|
| User registration | `POST /api/v1/auth/register` — validates username format, email format, password strength (uppercase+lowercase+digit, min 8 chars); rejects duplicate username/email with `409 USER_EXISTS` |
| Login | `POST /api/v1/auth/login` — OAuth2 password flow, returns access + refresh JWT pair; rate-limited to 5/minute per client |
| Token refresh | `POST /api/v1/auth/refresh` — exchanges a valid refresh token for a new access+refresh pair |
| Current user profile | `GET /api/v1/auth/me` — requires valid Bearer access token |
| Default role assignment | Every new user is auto-assigned the `viewer` role (created on first use) |
| RBAC-protected endpoints | `GET /api/v1/users`, `GET /api/v1/users/{id}` — require the `admin` role (or `is_superuser`); demonstrate `403 INSUFFICIENT_ROLE` otherwise |
| Pagination | `/users` supports `page` / `page_size` query params, returns a `{items, meta}` envelope |
| Health checks | `GET /health` (liveness), `GET /health/db` (readiness — verifies live MySQL connectivity) |

## 5–6. File Structure & Explanation of Every File

### `backend/app/config/settings.py`
`pydantic-settings`-based `Settings` class reading from environment/`.env`. Exposes
a cached `get_settings()` (via `lru_cache`) and computed properties
`sqlalchemy_database_uri` and `cors_origins_list`.

### `backend/app/database/base.py`
Declarative `Base` class (SQLAlchemy 2.0 `DeclarativeBase`) shared by every model.

### `backend/app/database/session.py`
Creates the SQLAlchemy `engine` (pool_pre_ping, pool_recycle=3600) and `SessionLocal`
factory; exposes the `get_db()` FastAPI dependency.

### `backend/app/models/mixins.py`
`TimestampMixin` — adds `created_at` (server default `now()`) and `updated_at`
(server default `now()`, `onupdate=now()`) to every table.

### `backend/app/models/*.py` (19 files)
One SQLAlchemy model per table — see §7/§8 below for full schema detail.

### `backend/app/models/__init__.py`
Imports every model module so `Base.metadata` is fully populated before Alembic
autogeneration or `create_all()` runs. This is the single most important file for
keeping migrations in sync — a model added without being imported here would be
silently invisible to Alembic.

### `backend/app/schemas/{user,role,token,common}.py`
Pydantic v2 schemas. `user.py` includes custom validators: username format
(`^[a-zA-Z0-9_.-]{3,50}$`) and password strength (upper+lower+digit, min 8 chars).
`common.py` defines the generic `PaginatedResponse[T]` and `ErrorResponse` envelopes
reused by every future list/error endpoint.

### `backend/app/repositories/base_repository.py`
Generic `BaseRepository[ModelType]` with `get_by_id`, `list`, `count`, `create`,
`delete` — implements the Repository Pattern so services never touch SQLAlchemy
`Session` query syntax directly.

### `backend/app/repositories/{user,role}_repository.py`
Concrete repositories adding entity-specific lookups (`get_by_username`,
`get_by_email`, `username_or_email_exists`, `get_or_create`).

### `backend/app/authentication/password_handler.py`
`hash_password` / `verify_password` using **bcrypt directly** (not passlib — passlib
1.7.4 has a known incompatibility with bcrypt ≥ 4.1 that emits spurious warnings/
errors; calling bcrypt directly avoids the dependency entirely).

### `backend/app/authentication/jwt_handler.py`
`create_access_token`, `create_refresh_token`, `decode_token` using `python-jose`.
Tokens carry `sub` (username), `iat`, `exp`, and `type` (`access`/`refresh`) claims;
decoding enforces the expected `type` so an access token cannot be replayed as a
refresh token or vice versa.

### `backend/app/authentication/dependencies.py`
`get_current_user`, `get_current_active_user`, and the `require_roles(*roles)`
dependency factory implementing RBAC (superusers bypass the check; otherwise the
user's role set must intersect the required roles).

### `backend/app/services/auth_service.py`
`AuthService.register/authenticate/login/refresh` — all business rules (duplicate
checks, default role assignment, credential verification, account-active check)
live here, independent of HTTP concerns.

### `backend/app/controllers/{auth,user}_controller.py`
Translate between schemas and services; the thin layer routers call into.

### `backend/app/routers/{auth,user,health}_router.py`
FastAPI `APIRouter`s: HTTP methods, status codes, `response_model`s, OpenAPI
`responses=` documentation, and dependency wiring (auth, RBAC, rate limiting).

### `backend/app/middleware/error_handler.py`
Registers exception handlers for `AppException` (and subclasses), FastAPI's
`RequestValidationError`, and a catch-all `Exception` handler — every error path
returns the same `{"error": {"code", "message"}}` shape. Uses `jsonable_encoder`
on validation errors specifically because Pydantic v2 embeds raw exception objects
in `ctx` that `json.dumps` cannot serialize directly (this was caught by the test
suite — see §40).

### `backend/app/middleware/logging_middleware.py`
`RequestLoggingMiddleware` — logs method, path, status code, duration per request.

### `backend/app/middleware/rate_limiter.py`
A single shared `slowapi.Limiter` instance (keyed by remote address), registered
onto `app.state` and wired to the `RateLimitExceeded` handler. Imported (not
re-instantiated) by any router that needs `@limiter.limit(...)`.

### `backend/app/main.py`
`create_app()` factory: configures logging, registers the rate limiter, exception
handlers, logging middleware, CORS, and mounts `health_router` (unprefixed),
`auth_router` and `user_router` (under `/api/v1`).

### `backend/alembic/env.py`
Alembic environment wired to `app.config.settings` (so the DB URL is never
duplicated) and `Base.metadata` (via `import app.models`), with `compare_type=True`
for both offline and online migration modes.

### `backend/alembic/versions/f5a5b68878ec_initial_schema.py`
The initial migration — **autogenerated by running `alembic revision --autogenerate`
against a live MySQL 8.0 container**, then applied with `alembic upgrade head` and
verified. Creates all 19 tables, the `user_roles` association table, every index,
and every unique/check constraint.

### `backend/tests/conftest.py`
Points the test suite at a dedicated `cloud_ai_platform_test` MySQL database
(forcibly overriding `MYSQL_DATABASE`, since it may already be set by the container
environment). Creates all tables once per session, and wraps each test in a
connection-level transaction that's rolled back afterwards for isolation — no
mocks, no SQLite substitution. Also resets the shared rate limiter before every
test (see §40).

### `database/schema/init.sql`
MySQL `docker-entrypoint-initdb.d` script: creates the `cloud_ai_platform_test`
database (the primary `cloud_ai_platform` DB is created automatically by the
`MYSQL_DATABASE` env var) and grants the app user privileges on it.

### `docker-compose.yml` / `backend/Dockerfile`
See §20 and §28–29.

## 7. Database Schema (Entity Overview)

19 tables + 1 association table, all `InnoDB`/`utf8mb4` (MySQL 8.0 defaults), every
table carrying `created_at`/`updated_at`:

| Table | Purpose | Key relationships |
|---|---|---|
| `users` | Platform accounts | 1—N `projects`, `api_keys`, `notifications`, `audit_logs`, `settings`; M—N `roles` |
| `roles` | Named permission groups | M—N `users` via `user_roles` |
| `user_roles` | Association table | PK(`user_id`, `role_id`) |
| `projects` | Top-level grouping | belongs to `users` (owner); 1—N `microservices`, `cloud_costs` |
| `microservices` | Deployable service | belongs to `projects`; 1—N `deployments` |
| `deployments` | K8s deployment of a microservice | belongs to `microservices`; 1—N `pods`, `metrics`, `resource_usage`, `predictions`, `anomaly_detections`, `failure_predictions`, `optimization_recommendations`, `alerts` |
| `pods` | Individual K8s pod | belongs to `deployments`; 1—N `metrics`, `failure_predictions` |
| `metrics` | Raw time-series metric points | optionally belongs to `deployments`/`pods` |
| `resource_usage` | Aggregated CPU/mem/disk/net snapshot | belongs to `deployments` |
| `predictions` | LSTM forecast output | belongs to `deployments`; `confidence_score` CHECKed to [0,1] |
| `anomaly_detections` | Isolation Forest output | belongs to `deployments` |
| `failure_predictions` | Random Forest output | belongs to `deployments`, optionally `pods`; `probability` CHECKed to [0,1] |
| `optimization_recommendations` | Optimization engine output | belongs to `deployments` |
| `alerts` | Threshold-triggered warnings | belongs to `deployments`; 1—N `notifications` |
| `notifications` | Per-user, per-channel delivery record | belongs to `users`, optionally `alerts` |
| `cloud_costs` | Billing records | belongs to `projects` |
| `audit_logs` | Compliance/security action trail | optionally belongs to `users` (SET NULL on delete) |
| `settings` | Per-user/global key-value config | optionally belongs to `users` |
| `api_keys` | Hashed long-lived credentials | belongs to `users` |
| `logs` | Structured application/service logs | standalone |

## 8. MySQL CREATE TABLE Scripts

The authoritative DDL is generated by Alembic (see §10), but expressed as raw SQL,
the core identity tables are equivalent to:

```sql
CREATE TABLE roles (
    id INTEGER NOT NULL AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    description VARCHAR(255),
    created_at DATETIME NOT NULL DEFAULT NOW(),
    updated_at DATETIME NOT NULL DEFAULT NOW() ON UPDATE NOW(),
    PRIMARY KEY (id),
    UNIQUE KEY ix_roles_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE users (
    id INTEGER NOT NULL AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(120) NOT NULL,
    full_name VARCHAR(120),
    hashed_password VARCHAR(255) NOT NULL,
    is_active BOOL NOT NULL,
    is_superuser BOOL NOT NULL,
    created_at DATETIME NOT NULL DEFAULT NOW(),
    updated_at DATETIME NOT NULL DEFAULT NOW() ON UPDATE NOW(),
    PRIMARY KEY (id),
    UNIQUE KEY uq_users_username (username),
    UNIQUE KEY uq_users_email (email),
    KEY ix_users_email_active (email, is_active)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE user_roles (
    user_id INTEGER NOT NULL,
    role_id INTEGER NOT NULL,
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE predictions (
    id INTEGER NOT NULL AUTO_INCREMENT,
    deployment_id INTEGER NOT NULL,
    model_type VARCHAR(30) NOT NULL,
    metric_type VARCHAR(50) NOT NULL,
    predicted_value FLOAT NOT NULL,
    confidence_score FLOAT NOT NULL,
    target_timestamp DATETIME NOT NULL,
    generated_at DATETIME NOT NULL,
    created_at DATETIME NOT NULL DEFAULT NOW(),
    updated_at DATETIME NOT NULL DEFAULT NOW() ON UPDATE NOW(),
    PRIMARY KEY (id),
    FOREIGN KEY (deployment_id) REFERENCES deployments(id) ON DELETE CASCADE,
    CONSTRAINT ck_prediction_confidence_range CHECK (confidence_score >= 0 AND confidence_score <= 1)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

The complete DDL for all 20 tables is in
`backend/alembic/versions/f5a5b68878ec_initial_schema.py` (generated, not
hand-written) and is applied verbatim by `alembic upgrade head`.

## 9. SQLAlchemy Models

All 19 models live in `backend/app/models/`, using SQLAlchemy 2.0's typed
`Mapped[...]`/`mapped_column(...)` style. Example (`Prediction`):

```python
class Prediction(TimestampMixin, Base):
    __tablename__ = "predictions"
    __table_args__ = (
        Index("ix_predictions_deployment_target", "deployment_id", "target_timestamp"),
        CheckConstraint("confidence_score >= 0 AND confidence_score <= 1",
                         name="ck_prediction_confidence_range"),
    )
    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    deployment_id: Mapped[int] = mapped_column(
        Integer, ForeignKey("deployments.id", ondelete="CASCADE"), nullable=False, index=True
    )
    model_type: Mapped[str] = mapped_column(String(30), nullable=False, default="lstm")
    metric_type: Mapped[str] = mapped_column(String(50), nullable=False)
    predicted_value: Mapped[float] = mapped_column(Float, nullable=False)
    confidence_score: Mapped[float] = mapped_column(Float, nullable=False)
    target_timestamp: Mapped[datetime] = mapped_column(DateTime, nullable=False)
    generated_at: Mapped[datetime] = mapped_column(DateTime, nullable=False)

    deployment: Mapped["Deployment"] = relationship("Deployment", back_populates="predictions")
```

-- Find a user by username (used by login)
SELECT * FROM users WHERE username = 'jdoe';

-- Check for duplicate username/email at registration
SELECT * FROM users WHERE username = 'jdoe' OR email = 'jdoe@example.com';

## 12–15. API Endpoints, Request/Response/Error Payloads

All endpoints are also live and interactively documented at `/docs` (Swagger UI)
and `/redoc` once the backend is running.

### `POST /api/v1/auth/register`
Request:
```json
{ "username": "jdoe", "email": "jdoe@example.com", "full_name": "Jane Doe", "password": "Sup3rSecret1" }
```
Response `201`:
```json
{ "id": 1, "username": "jdoe", "email": "jdoe@example.com", "full_name": "Jane Doe",
  "is_active": true, "is_superuser": false,
  "roles": [{ "id": 1, "name": "viewer", "description": "Read-only access to dashboards and reports" }] }
```
Errors: `409 USER_EXISTS`, `422 VALIDATION_ERROR` (weak password / malformed username or email)

### `POST /api/v1/auth/login` (form-encoded, OAuth2 password flow)
Request (form fields): `username=jdoe&password=Sup3rSecret1`
Response `200`:
```json
{ "access_token": "<jwt>", "refresh_token": "<jwt>", "token_type": "bearer" }
```
Errors: `401 INVALID_CREDENTIALS`, `401 INACTIVE_USER`, `429 Too Many Requests` (rate limit)

### `POST /api/v1/auth/refresh?refresh_token=<jwt>`
Response `200`: same shape as login. Errors: `401 INVALID_TOKEN`, `401 INVALID_TOKEN_TYPE`, `401 USER_NOT_FOUND`

### `GET /api/v1/auth/me` (Bearer token required)
Response `200`: same `UserRead` shape as register. Error: `401 Unauthorized`

### `GET /api/v1/users?page=1&page_size=20` (admin role required)
Response `200`:
```json
{ "items": [ { "id": 1, "username": "jdoe", "...": "..." } ],
  "meta": { "total": 1, "page": 1, "page_size": 20, "total_pages": 1 } }
```
Error: `403 INSUFFICIENT_ROLE`

### `GET /health` / `GET /health/db`
`{"status": "ok"}` / `{"status": "ok", "database": "reachable"}`

### Global error envelope
Every error response, regardless of source, has the shape:
```json
{ "error": { "code": "SOME_CODE", "message": "Human readable message" } }
```
(validation errors additionally carry a `details` array of field-level issues)

## 16. Authentication Flow

1. Client `POST /auth/register` → account created with bcrypt-hashed password, default `viewer` role.
2. Client `POST /auth/login` (username+password) → server verifies via `AuthService.authenticate` → issues a short-lived access token (30 min default) and a longer-lived refresh token (7 days default), both HS256 JWTs signed with `SECRET_KEY`.
3. Client calls protected endpoints with `Authorization: Bearer <access_token>`. `get_current_user` decodes the token (validating `type == access`), loads the user by `sub` (username).
4. RBAC-protected endpoints additionally run `require_roles(...)`, which checks `is_superuser` or role-name intersection.
5. When the access token expires, the client calls `POST /auth/refresh` with the refresh token to obtain a new pair without re-entering credentials.

## 40. Troubleshooting / Common Errors Encountered This Phase

These are real issues hit and fixed while building this phase, kept here since they
will recur for anyone reproducing the setup:

1. **`ImportError: email-validator is not installed`** — Pydantic's `EmailStr` requires the optional `email-validator` package, which isn't pulled in by `pydantic` alone. Fix: added `pydantic[email]==2.10.4` to `requirements.txt`.
2. **`TypeError: Object of type ValueError is not JSON serializable`** on `422` responses — a `field_validator` raising a plain `ValueError` (e.g. weak-password check) causes FastAPI's `RequestValidationError.errors()` to embed the raw exception in `ctx.error`, which `json.dumps` can't handle. Fix: run `exc.errors()` through `fastapi.encoders.jsonable_encoder` before returning it in the custom exception handler.
3. **Tests intermittently returning `429` on login** — `slowapi`'s in-memory rate-limit counters persist for the lifetime of the process, so a test module making many login calls (across many tests) eventually trips the same 5/minute budget used in production. Fix: an autouse `_reset_rate_limiter` fixture calls `limiter.reset()` before every test.
4. **Docker Desktop engine not running** ("open //./pipe/dockerDesktopLinuxEngine") — Docker Desktop was installed but not started. Fix: launch `Docker Desktop.exe` and wait ~30–60s for the engine to initialize before running `docker compose`.
5. **Host Python interpreter broken** — `py -0p` reported Python 3.13 at a path that didn't exist on disk, and `python.exe` resolved to a non-functional Windows Store alias stub. Rather than fight the host environment, all Python execution (Alembic, pytest) was run **inside the backend Docker container** instead, with the `backend/` directory bind-mounted so generated files (migrations) land back on the host. This is arguably more representative of how the project will actually run in CI/production anyway.

## 41. Performance Optimizations

- `pool_pre_ping=True` + `pool_recycle=3600` on the SQLAlchemy engine to avoid stale-connection errors after MySQL idle timeouts.
- Composite indexes added on every high-cardinality lookup pattern anticipated by later phases (e.g. `ix_metrics_lookup(deployment_id, metric_type, recorded_at)`, `ix_predictions_deployment_target(deployment_id, target_timestamp)`) rather than only single-column indexes.
- `lazy="selectin"` on `User.roles` to avoid N+1 queries when serializing a user's roles.

## 42. Security Measures

- Passwords hashed with bcrypt (12 rounds), never stored or logged in plaintext.
- JWTs signed with HS256 and a configurable `SECRET_KEY`; access and refresh tokens carry a `type` claim so one cannot be used as the other.
- Rate limiting on `/auth/login` (5/minute per client) to slow brute-force credential guessing.
- RBAC enforced server-side via `require_roles()`, not trusted from client input.
- Consistent error envelope avoids leaking stack traces; the catch-all `Exception` handler returns a generic `500` message.
- `.env` files are git-ignored; only `.env.example` templates are committed.
- Input validation at the boundary (Pydantic schemas: username pattern, email format, password strength) before anything reaches the service layer.
- CORS explicitly configured (not wildcard-by-default in a way that ignores credentials).

## 48. Complete Explanation of Every Command Executed To Build This Phase

| Command | Purpose |
|---|---|
| `docker compose up -d mysql` | Start MySQL 8.0 in the background, running `database/schema/init.sql` on first boot to create the test database |
| `docker compose build backend` | Build the FastAPI backend image from `backend/Dockerfile` |
| `docker compose run --rm -v "<path>:/app" backend alembic revision --autogenerate -m "initial schema"` | Compare `Base.metadata` (all 19 models) against the live (empty) MySQL schema and generate the migration file; bind-mount so the generated file lands on the host, not just inside the ephemeral container |
| `docker compose run --rm -v "<path>:/app" backend alembic upgrade head` | Apply that migration, creating all tables/indexes/constraints |
| `docker compose run --rm -v "<path>:/app" backend pytest -v` | Run the full test suite inside the same environment the app runs in |

| `Invoke-RestMethod` / `Invoke-WebRequest` (PowerShell) | Manual end-to-end HTTP verification of `/health`, `/health/db`, `/openapi.json`, `/docs`, `/auth/register`, `/auth/login`, `/auth/me`, `/users` (RBAC denial), and weak-password validation |
